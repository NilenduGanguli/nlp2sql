# Oracle → Knowledge Graph Pipeline

This document explains how KnowledgeQL extracts metadata from an Oracle database, converts it into an in-memory property graph, and makes it available for NLP-to-SQL query generation.

---

## Overview

```
Oracle DB
   │
   ▼
OracleMetadataExtractor       ← 15 SQL queries across data dictionary views
   │  OracleMetadata
   ▼
GraphBuilder                  ← 13 build steps: nodes + edges + algorithms
   │  KnowledgeGraph (in-memory)
   ▼
InferredGlossaryBuilder       ← derives BusinessTerm nodes from metadata
   │
   ▼
KnowledgeGraph (ready)        ← used by traversal.py for all query functions
```

No external graph database is used. Everything lives in Python dicts inside the `KnowledgeGraph` class.

## Stage 1 — Configuration

**Files:** `knowledge_graph/config.py`

Two dataclasses control the entire pipeline:

### `OracleConfig`

| Field | Env Var | Default | Description |
|---|---|---|---|
| `dsn` | `ORACLE_DSN` | — | Oracle connect string, e.g. `localhost:1521/FREEPDB1` |
| `user` | `ORACLE_USER` | — | DB username |
| `password` | `ORACLE_PASSWORD` | — | DB password |
| `target_schemas` | `ORACLE_TARGET_SCHEMAS` | (all accessible) | Comma-separated list of schema names to introspect |
| `sample_rows` | `ORACLE_SAMPLE_ROWS` | `10` | Rows sampled per table for `sample_values` on columns |
| `use_dba_views` | `ORACLE_USE_DBA_VIEWS` | `true` | Use `DBA_*` views (requires DBA privilege); `false` uses `ALL_*` views |

The `view_prefix` property returns `"DBA"` or `"ALL"` and is used to prefix every data dictionary view name in every SQL query.

### `GraphConfig`

| Field | Env Var | Default | Description |
|---|---|---|---|
| `oracle` | — | — | Composed `OracleConfig` |
| `max_join_path_hops` | `MAX_JOIN_PATH_HOPS` | `4` | Max FK hops considered when computing JOIN_PATH edges |
| `similarity_levenshtein_max` | `SIMILARITY_LEVENSHTEIN_MAX` | `2` | Max edit distance for SIMILAR_TO edges |
| `similarity_min_score` | `SIMILARITY_MIN_SCORE` | `0.75` | Min normalized score `1 - dist/max_len` for SIMILAR_TO |

## Stage 2 — Oracle Metadata Extraction

**File:** `knowledge_graph/oracle_extractor.py`
**Class:** `OracleMetadataExtractor`
**Entry point:** `extractor.extract() → OracleMetadata`

Connects to Oracle via `python-oracledb` (thin mode — no Instant Client needed), runs 15 SQL queries against the data dictionary, and returns an `OracleMetadata` dataclass.

### Bind parameter convention

All SQL queries use **named bind variables** `:s0, :s1, ...` for the schema list, generated by:

```python
def _in_clause(schemas, column) -> str:
    # → "column IN (:s0, :s1, ...)"

def _bind_schemas(schemas) -> dict:
    # → {"s0": "SCHEMA1", "s1": "SCHEMA2", ...}
```

The same bind dict reuses the same names across multiple `IN` clauses in the same query.

---

### 2.1 Tables

**View:** `DBA_TABLES` + `DBA_TAB_COMMENTS`

```sql
SELECT
    t.owner, t.table_name,
    t.num_rows, t.avg_row_len,
    TO_CHAR(t.last_analyzed, 'YYYY-MM-DD HH24:MI:SS'),
    NVL(t.iot_type, 'TABLE') AS table_type,
    t.partitioned, t.temporary,
    tc.comments
FROM DBA_TABLES t
LEFT JOIN DBA_TAB_COMMENTS tc
    ON  tc.owner = t.owner
    AND tc.table_name = t.table_name
    AND tc.table_type = 'TABLE'
WHERE t.owner IN (:s0, :s1, ...)
  AND (t.iot_type IS NULL OR t.iot_type != 'IOT_OVERFLOW')
ORDER BY t.owner, t.table_name
```

**Produces:** `List[TableNode]`
Each `TableNode` carries: `schema`, `name`, `row_count`, `avg_row_length`, `last_analyzed`, `table_type`, `partitioned`, `temporary`, `comments`.
FQN: `SCHEMA.TABLE`

---

### 2.2 Columns

**View:** `DBA_TAB_COLUMNS` + `DBA_COL_COMMENTS` + `DBA_TAB_COL_STATISTICS`

```sql
SELECT
    c.owner, c.table_name, c.column_name,
    c.data_type, c.data_length, c.data_precision, c.data_scale,
    c.nullable, c.data_default, c.column_id,
    cc.comments,
    s.num_distinct, s.histogram
FROM DBA_TAB_COLUMNS c
LEFT JOIN DBA_COL_COMMENTS cc ON (owner/table/column match)
LEFT JOIN DBA_TAB_COL_STATISTICS s ON (owner/table/column match)
WHERE c.owner IN (:s0, ...)
ORDER BY c.owner, c.table_name, c.column_id
```

**Produces:** `List[ColumnNode]`
Each `ColumnNode` carries: `schema`, `table_name`, `name`, `data_type`, `data_length`, `precision`, `scale`, `nullable`, `default_value`, `column_id`, `comments`, `num_distinct`, `histogram_type`.
FQN: `SCHEMA.TABLE.COLUMN`

Boolean flags `is_pk`, `is_fk`, `is_indexed` are set in the post-processing step after all constraints and indexes are extracted.

---

### 2.3 Primary Keys

**View:** `DBA_CONSTRAINTS` + `DBA_CONS_COLUMNS`

```sql
SELECT
    a.owner, a.table_name, a.constraint_name,
    b.column_name, b.position
FROM DBA_CONSTRAINTS a
JOIN DBA_CONS_COLUMNS b ON (owner/constraint match)
WHERE a.owner IN (:s0, ...)
  AND a.constraint_type = 'P'
  AND a.status = 'ENABLED'
ORDER BY a.owner, a.table_name, b.position
```

**Produces:** `List[HasPrimaryKeyRel]` — each records `table_fqn`, `column_fqn`, `constraint_name`.

---

### 2.4 Foreign Keys

**View:** `DBA_CONSTRAINTS` + `DBA_CONS_COLUMNS` (self-joined four times)

```sql
SELECT
    a.owner, a.table_name, a.constraint_name,
    NVL(a.delete_rule, 'NO ACTION'),
    b.column_name   AS fk_column,  b.position,
    c.owner         AS ref_owner,  c.table_name AS ref_table,
    d.column_name   AS ref_column
FROM DBA_CONSTRAINTS a
JOIN DBA_CONS_COLUMNS b ON (a.owner / a.constraint_name)
JOIN DBA_CONSTRAINTS  c ON (c.constraint_name = a.r_constraint_name)
JOIN DBA_CONS_COLUMNS d ON (d.owner/constraint/position = b.position)
WHERE a.owner IN (:s0, ...)
  AND a.constraint_type = 'R'
  AND a.status = 'ENABLED'
ORDER BY a.owner, a.table_name, a.constraint_name, b.position
```

**Produces:** `List[HasForeignKeyRel]` — each records `source_column_fqn`, `target_column_fqn`, `constraint_name`, `on_delete_action`.

---

### 2.5 Indexes

**Views:** `DBA_INDEXES` + `DBA_IND_COLUMNS` (two queries)

First fetches index metadata (name, type, uniqueness, tablespace, compression), then all indexed columns sorted by `column_position`. Column names are joined into a CSV string and stored as `IndexNode.columns_list`.

**Produces:** `List[IndexNode]`
Each `IndexNode` carries: `name`, `schema`, `table_name`, `index_type`, `uniqueness`, `columns_list`, `tablespace`, `compression`.

---

### 2.6 Constraints

**View:** `DBA_CONSTRAINTS`

Extracts `P` (primary key), `R` (foreign key), `U` (unique), and `C` (check) constraints. Excludes `SYS_`-prefixed system constraints. The `search_condition` for CHECK constraints is stored (max 500 chars).

**Produces:** `List[ConstraintNode]`

---

### 2.7 Views & Materialized Views

Uses `DBMS_METADATA.GET_DDL` to retrieve full view DDL, falling back to `SUBSTR(text, 1, 4000)` if that fails. Materialized views are fetched from `DBA_MVIEWS` with refresh mode and last refresh date.

View-to-table dependencies come from `DBA_DEPENDENCIES`:

```sql
SELECT d.owner, d.name, d.referenced_owner, d.referenced_name, d.referenced_type
FROM DBA_DEPENDENCIES d
WHERE d.owner IN (:s0, ...)
  AND d.type = 'VIEW'
  AND d.referenced_type IN ('TABLE', 'VIEW', 'MATERIALIZED VIEW')
```

**Produces:** `List[ViewNode]`, `Dict[view_fqn, List[dep_dict]]`

---

### 2.8 Procedures / Functions / Packages

**View:** `DBA_PROCEDURES` LEFT JOIN `DBA_OBJECTS`

> **Note:** `ALL_PROCEDURES` in Oracle 23c has no `STATUS` column.
> The extractor always joins with `DBA_OBJECTS` (or `ALL_OBJECTS`) to retrieve status.

```sql
SELECT DISTINCT
    p.owner, p.object_name, p.object_type,
    NVL(o.status, 'VALID') AS status
FROM DBA_PROCEDURES p
LEFT JOIN DBA_OBJECTS o
    ON  o.owner = p.owner
    AND o.object_name = p.object_name
    AND o.object_type = p.object_type
WHERE p.owner IN (:s0, ...)
  AND p.object_type IN ('PROCEDURE', 'FUNCTION', 'PACKAGE')
```

**Produces:** `List[ProcedureNode]`

---

### 2.9 Synonyms

**View:** `DBA_SYNONYMS`

Both the owner filter and the `table_owner` filter use the same named bind dict (the two `IN` clauses reference the same `:s0, :s1, ...` variables), so only one call to `_bind_schemas()` is needed.

**Produces:** `List[SynonymNode]`

---

### 2.10 Sequences

**View:** `DBA_SEQUENCES`

Captures `min_value`, `max_value`, `increment_by`, `cache_size`.

**Produces:** `List[SequenceNode]`

---

### 2.11 Sample Data

Per table (up to `config.sample_rows` rows):

```sql
SELECT * FROM "SCHEMA"."TABLE" FETCH FIRST N ROWS ONLY
```

String values longer than 200 chars are truncated. Up to 5 non-null values per column are stored on `ColumnNode.sample_values` and are later used by the glossary builder to enrich business term definitions with valid enum values.

---

### 2.12 Post-Processing

After all queries complete, three steps run synchronously:

1. **`_flag_columns(meta)`** — builds sets of PK FQNs, FK FQNs, and indexed FQNs, then sets `is_pk`, `is_fk`, `is_indexed` booleans on each `ColumnNode`.
2. **`_build_indexed_column_map(indexes)`** — builds `{owner: {table: [col, ...]}}` for O(1) index lookups.
3. **`_attach_sample_data(meta)`** — assigns `sample_data` lists to `TableNode` objects and propagates `sample_values` to `ColumnNode` objects.

## Stage 3 — Graph Construction

**File:** `knowledge_graph/graph_builder.py`
**Class:** `GraphBuilder`
**Entry point:** `builder.build(metadata) → Dict[str, int]`

Takes the `OracleMetadata` snapshot and calls 13 methods in sequence. Every object is written to `self.graph` (a `KnowledgeGraph` instance) via `merge_node()` and `merge_edge()`.

### Node and edge label reference

| Build step | Node label | Edge type(s) |
|---|---|---|
| `_build_schemas` | `Schema` | — |
| `_build_tables` | `Table` | `BELONGS_TO` (Table → Schema) |
| `_build_columns` | `Column` | `HAS_COLUMN` (Table → Column) |
| `_build_primary_keys` | — | `HAS_PRIMARY_KEY` (Table → Column) |
| `_build_foreign_keys` | — | `HAS_FOREIGN_KEY` (Column → Column) |
| `_build_indexes` | `Index` | `HAS_INDEX` (Table → Index), `INDEXED_BY` (Column → Index) |
| `_build_constraints` | `Constraint` | `HAS_CONSTRAINT` (Table → Constraint) |
| `_build_views` | `View` | `BELONGS_TO` (View → Schema), `DEPENDS_ON` (View → Table) |
| `_build_procedures` | `Procedure` | `BELONGS_TO` (Procedure → Schema) |
| `_build_synonyms` | `Synonym` | — |
| `_build_sequences` | `Sequence` | `BELONGS_TO` (Sequence → Schema) |
| `_compute_join_paths` | — | `JOIN_PATH` (Table ↔ Table, bidirectional) |
| `_compute_similar_to` | — | `SIMILAR_TO` (Column ↔ Column, bidirectional) |

---

### 3.1 JOIN_PATH Algorithm

This is the most important computed edge type — it lets the SQL generator know how to join any two tables.

**Algorithm:**

```
1. Build a NetworkX MultiDiGraph G.
   For each HasForeignKeyRel FK:
     G.add_edge(FK.src_table, FK.tgt_table, src_col=..., tgt_col=..., constraint=...)
     G.add_edge(FK.tgt_table, FK.src_table, src_col=..., tgt_col=...,  # reverse
                constraint=FK.constraint_name + "_REV")

2. For every unique (table_A, table_B) pair:
     path_nodes = nx.shortest_path(G.to_undirected(), A, B)

3. If len(path_nodes) - 1 > max_join_path_hops: skip.

4. Walk each consecutive pair (u, v) in path_nodes.
   For each hop, call _best_edge(G, u, v) to pick the lowest-key
   parallel edge. Record: {"src": col_u, "tgt": col_v, "constraint": ...}

5. Write two JOIN_PATH edges to the graph (merge_key="path_key"):
     A → B  with join_columns=[hop1, hop2, ...], weight=hops
     B → A  with join_columns reversed (src/tgt flipped)
```

- `weight` = number of FK hops (1 for a direct FK, 2 for A→B→C, etc.)
- `join_type` = `"INNER"` (all FK joins)
- `cardinality` = `"N:1"` for forward direction, `"1:N"` for reverse

**Example:** CUSTOMERS → TRANSACTIONS (2 hops via ACCOUNTS):
```json
{
  "join_columns": [
    {"src": "KYC.CUSTOMERS.CUSTOMER_ID", "tgt": "KYC.ACCOUNTS.CUSTOMER_ID",      "constraint": "FK_ACCOUNTS_CUSTOMER"},
    {"src": "KYC.ACCOUNTS.ACCOUNT_ID",   "tgt": "KYC.TRANSACTIONS.ACCOUNT_ID",   "constraint": "FK_TRANSACTIONS_ACCOUNT"}
  ],
  "join_type": "INNER",
  "cardinality": "N:1",
  "weight": 2,
  "source": "precomputed"
}
```

---

### 3.2 SIMILAR_TO Algorithm

Connects semantically related columns across different tables to help the SQL generator find join candidates that aren't declared as FKs.

Three strategies run in priority order, using a `seen` set to avoid duplicate edges:

| Priority | Strategy | Score | Match Type | Condition |
|---|---|---|---|---|
| 1 | Exact name | 1.0 | `"exact"` | Same column name in ≥2 different tables |
| 2 | FK suffix pattern | 0.9 | `"suffix"` | Name ends with `_ID`, `_CODE`, `_KEY`, `_NO`, `_NUM`, or `_REF` |
| 3 | Levenshtein distance | computed | `"levenshtein"` | Edit distance ≤ `max` AND normalized score ≥ `min` |

Levenshtein score formula: `score = 1 - edit_distance / max(len(name1), len(name2))`

## Stage 4 — Glossary Inference

**File:** `knowledge_graph/glossary_loader.py`
**Class:** `InferredGlossaryBuilder`
**Entry point:** `builder.build(metadata) → {"terms": N, "mappings": M}`

Runs after the graph is built. Mines the `OracleMetadata` to create `BusinessTerm` nodes and `MAPS_TO` edges that allow the query engine to resolve plain-language terms like *"risk rating"* or *"PEP status"* to specific columns.

### Sensitivity classification

Each column name is tokenized on `_` and matched against keyword sets:

| Level | Example column names |
|---|---|
| `RESTRICTED` | `SSN`, `PASSPORT_NO`, `PASSWORD`, `DATE_OF_BIRTH` |
| `CONFIDENTIAL` | `RISK_RATING`, `BALANCE`, `AMOUNT`, `PEP_TYPE`, `NATIONALITY` |
| `INTERNAL` | Everything else |

### Term humanization

`UPPER_SNAKE_CASE` column/table names are converted to readable labels:
- Known abbreviations (`KYC`, `PEP`, `AML`, `SSN`, `DOB`, …) stay uppercase.
- All other tokens are title-cased.

Examples: `RISK_RATING` → `"Risk Rating"`, `KYC_REVIEW_DATE` → `"KYC Review Date"`, `CUSTOMER_ID` → `"Customer ID"`

### Definition confidence scoring

| Source | Confidence | Definition text |
|---|---|---|
| Column comment exists | **0.95** | Comment text verbatim |
| Table comment exists, no column comment | **0.65** | `"{term} in {table comment}"` |
| No comments | **0.50** | `"{term} in SCHEMA.TABLE"` |
| Table-level term (from table comment) | **0.80** | Table comment verbatim |

If sample values exist and cardinality is low (≤ 30 distinct values), the definition is enriched:
`". Valid values: LOW, MEDIUM, HIGH, VERY_HIGH"` — and confidence is bumped to at least `0.65`.

### Deduplication

When multiple columns produce the same humanized term label (e.g. `RISK_RATING` appears in both `CUSTOMERS` and `RISK_ASSESSMENTS`), the one with the **highest confidence** wins. All mappings are still written as `MAPS_TO` edges pointing to their respective FQNs, so a term can map to multiple columns.

### Structural columns skipped

Pure structural column names like `ID`, `SEQ`, `CREATED_AT`, `UPDATED_BY`, `ROW_VERSION`, etc. are skipped — they don't carry business meaning on their own.

### Output

```
BusinessTerm node (e.g. "Risk Rating")
  │
  MAPS_TO (confidence=0.95) → KYC.CUSTOMERS.RISK_RATING
  MAPS_TO (confidence=0.95) → KYC.RISK_ASSESSMENTS.RISK_LEVEL
```

For the KYC schema, this produces 52 `BusinessTerm` nodes and 62 `MAPS_TO` edges.

## Stage 5 — Graph Storage

**File:** `knowledge_graph/graph_store.py`
**Class:** `KnowledgeGraph`

### Internal data structures

```python
_nodes:   defaultdict(dict)          # label → {node_id → props_dict}
_edges:   defaultdict(list)          # rel_type → [edge_dicts]
_out_idx: defaultdict(defaultdict)   # rel_type → {from_id → [edge_dicts]}
_in_idx:  defaultdict(defaultdict)   # rel_type → {to_id   → [edge_dicts]}
```

All four structures are kept in sync on every write. Node and edge operations are O(1) average-case hash lookups.

### Node operations

```python
graph.merge_node(label, node_id, props)
# Upserts: if node exists, existing props are updated (not replaced).
# node_id is always the FQN string (e.g. "KYC.CUSTOMERS").

graph.get_node(label, node_id) → dict | None
graph.get_all_nodes(label) → List[dict]
graph.count_nodes(label) → int
graph.set_node_prop(label, node_id, key, value)   # mutate single prop
```

### Edge operations

```python
graph.merge_edge(rel_type, from_id, to_id, merge_key=None, **props)
# With no merge_key:  at most one edge per (from_id, to_id) pair.
# With merge_key="x": at most one edge per (from_id, to_id, props["x"]) triple.
# If a matching edge exists its props are updated in place.

graph.get_out_edges(rel_type, from_id) → List[dict]
graph.get_in_edges(rel_type, to_id) → List[dict]
graph.get_all_edges(rel_type) → List[dict]
graph.count_edges(rel_type) → int
```

Each edge dict always contains `_from` and `_to` plus any user-supplied kwargs.

## Stage 6 — Orchestration

**File:** `knowledge_graph/init_graph.py`
**Function:** `initialize_graph(config=None, refresh_only=False) → (KnowledgeGraph, report)`

Runs all five stages in sequence and returns the ready-to-query graph plus a structured report.

```
initialize_graph()
  │
  ├─ 1. OracleMetadataExtractor.check_connectivity()   → fail-fast
  ├─ 2. extractor.extract()                            → OracleMetadata
  ├─ 3. GraphBuilder(config).build(metadata)           → KnowledgeGraph + stats
  ├─ 4. InferredGlossaryBuilder(graph).build(metadata) → BusinessTerm nodes
  ├─ 5. validate_graph(graph)                          → 4 consistency checks
  └─ returns (graph, report)
```

### Graph validation checks

| Check | Condition |
|---|---|
| `Total tables > 0` | `count_nodes("Table") >= 1` |
| `Total columns > 0` | `count_nodes("Column") >= 1` |
| `HAS_COLUMN edges exist` | `count_edges("HAS_COLUMN") >= 1` |
| `Orphan columns == 0` | Every Column node has an incoming `HAS_COLUMN` edge |

## How it all connects to query generation

When a user types a natural language query, the agent pipeline uses the graph like this:

1. **Entity extraction** — LLM extracts table hints and business terms from the query.
2. **`search_schema(graph, hint)`** — text-searches `Table` and `Column` node names/comments to find matching FQNs.
3. **`resolve_business_term(graph, term)`** — walks `MAPS_TO` edges to find the column FQNs behind a term like *"risk rating"*.
4. **`get_context_subgraph(graph, fqns)`** — collects a table's columns, FKs, indexes, and business terms from the graph.
5. **`serialize_context_to_ddl(context)`** — renders the subgraph as DDL-style text injected into the LLM prompt.
6. **`find_join_path(graph, t1, t2)`** — looks up the pre-computed `JOIN_PATH` edge (or runs BFS on the fly) to tell the LLM exactly which columns to join on.
7. **LLM generates SQL** — using the DDL context and join hints.
8. **`get_index_hints(graph, col_fqns)`** — supplies `INDEXED_BY` edges so the LLM can add `/*+ INDEX(...) */` hints.
